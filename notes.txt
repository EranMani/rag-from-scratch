PHASE 0 - Separation of concerts

* Each folder owns excatly one responsibility
* Never touch unrelated code

rag-from-scratch/
│
├── src/
│   ├── app/      ← HTTP + UI (presentation)
│   └── rag/      ← Business logic (pipeline)
├── data/         ← Persistent content (state)
├── monitoring/   ← Infrastructure config (observability)
└── scripts/      ← Operator tools (devops)

* Each layer only talks to the layer below it
	- app => rag
	- rag => data
	- monitoring => observes app

* Nothing goes sideways or upward


* src/ layout => app/ and rag/ live inside src/, not at the root
	- Follows modern Python packaging standards (PyPA recommendation)
	- ENV PYTHONPATH=/app/src in Dockerfile tells Python where to find packages
	- All imports stay unchanged (from app.core.config import settings)
	- data/, monitoring/, scripts/ stay at root — they are not Python packages

* src/app/ => public face of the system. 
	- HTTP
	- ROUTING : fastapi endpoints. thin, no business logic here
	- UI : nicegui chat interface. A presentation concern, not pipeline
	- CORE : config, logging and metrics. everything the app needs to run
* src/rag/ => brain of the system. Business logic with no HTTP concerns
	- pipeline : core RAG steps in sequence. each file is one steps
	- embeddings : capability that multiple pipeline steps use
	- providers : LLM abstraction (openai and ollama). switching providers is one line change
	- cache : redis, caching is a pipeline optimization. wraps retrieval and generation steps
	- resilience : circuit breakers and bm25 fallback. pipeline concern
	- memory : conversation state and user profiles. pipeline layer, feeds context into the generator
	- chain : imports all the others and assembles the full pipeline. a conductor
	
* /data => persistent storage lives outside the code
	- knowledge_base : content, the markdown files the system indexes
	- chroma_db : on-disk persistence. generated at runtime. re-built by ingest.py
	
* /monitoring => config files for three monitoring tools, YAML & JSON
	- PROMETHEUS
	- GRAFANA
	- LOGSTASH
	
* /scripts => commands that a developer or operator runs manually
	- INGESTING DOCUMENTS
	- RUNNING CONCEPT DEMOS
	
	
PHASE 1

* pyproject.toml => modern python equivalent of requirements.txt, but more powerful
	- Defines what the project is (name, requires-python version)
	- Dependencies are split intentionally
		* production deps go under project, dev-only tools go under tool.uv
		* running --no-dev means test libraries never end up in the production image
	- Version control, gives freedom to resolve compatiable versions.
	- UV.LOCK : a file uv generated that pins everything precisely, this file is what we commit in the end
	
* .env.example => goes to github with no real secrets, just placeholder values
	- Documents every variable the system needs
	- the .env file contains secrets (api keys) and is always gitignored
	- Contract between the codebase and the environment
	- When adding new var to .env, we must update the .env.example as well to be documented for other developers
	
* DockerFile => package the os layer, python version, all dependencies and config into a single unit called container
	- Docker image is a recipe, a frozen snapshot of an environment
	- Container is a running instance of that recipe, like process spawned from the image
	- Can run 10 containers from one image simultaneously
	- In this project, 9 services needs to talk to each other:
		* app -> charoma db address
		* grafana -> prometheus address
		* logstash -> elasticsearch address
	- Layer caching : each instruction (RUN, COPY, FROM) creates a layer.
		* docker caches each layer and only rebuilds from the first changed layer downward
		* dependencies change far less often than the source code
		* COPY source code is last because this changes most often, so it belongs at the bottom
		* EXPOSE is documentation, tells docker which port the container listens on
		* CMD is default command when the container starts
	- Order dockerfile instructions from least-likely-to-change to most-likely-to-change.
		* maximizes cache hits and keeps rebuild times fast
