PHASE 0 - Separation of concerns

* Each folder owns exactly one responsibility
* Never touch unrelated code

rag-from-scratch/
│
├── src/
│   ├── app/      ← HTTP + UI (presentation)
│   └── rag/      ← Business logic (pipeline)
├── data/         ← Persistent content (state)
├── monitoring/   ← Infrastructure config (observability)
└── scripts/      ← Operator tools (devops)

* Each layer only talks to the layer below it
	- app => rag
	- rag => data
	- monitoring => observes app

* Nothing goes sideways or upward


* src/ layout => app/ and rag/ live inside src/, not at the root
	- Follows modern Python packaging standards (PyPA recommendation)
	- ENV PYTHONPATH=/app/src in Dockerfile tells Python where to find packages
	- All imports stay unchanged (from app.core.config import settings)
	- data/, monitoring/, scripts/ stay at root — they are not Python packages

* src/app/ => public face of the system. 
	- HTTP
	- ROUTING : fastapi endpoints. thin, no business logic here
	- UI : nicegui chat interface. A presentation concern, not pipeline
	- CORE : config, logging and metrics. everything the app needs to run
* src/rag/ => brain of the system. Business logic with no HTTP concerns
	- pipeline : core RAG steps in sequence. each file is one steps
	- embeddings : capability that multiple pipeline steps use
	- providers : LLM abstraction (openai and ollama). switching providers is one line change
	- cache : redis, caching is a pipeline optimization. wraps retrieval and generation steps
	- resilience : circuit breakers and bm25 fallback. pipeline concern
	- memory : conversation state and user profiles. pipeline layer, feeds context into the generator
	- chain : imports all the others and assembles the full pipeline. a conductor
	
* /data => persistent storage lives outside the code
	- knowledge_base : content, the markdown files the system indexes
	- chroma_db : on-disk persistence. generated at runtime. re-built by ingest.py
	
* /monitoring => config files for three monitoring tools, YAML & JSON
	- PROMETHEUS
	- GRAFANA
	- LOGSTASH
	
* /scripts => commands that a developer or operator runs manually
	- INGESTING DOCUMENTS
	- RUNNING CONCEPT DEMOS
	
	
PHASE 1

* pyproject.toml => modern python equivalent of requirements.txt, but more powerful
	- Defines what the project is (name, requires-python version)
	- Dependencies are split intentionally
		* production deps go under project, dev-only tools go under tool.uv
		* running --no-dev means test libraries never end up in the production image
	- Version control, gives freedom to resolve compatiable versions.
	- UV.LOCK : a file uv generated that pins everything precisely, this file is what we commit in the end
	
* .env.example => goes to github with no real secrets, just placeholder values
	- Documents every variable the system needs
	- the .env file contains secrets (api keys) and is always gitignored
	- Contract between the codebase and the environment
	- When adding new var to .env, we must update the .env.example as well to be documented for other developers
	
* DockerFile => package the os layer, python version, all dependencies and config into a single unit called container
	- Docker image is a recipe, a frozen snapshot of an environment
	- Container is a running instance of that recipe, like process spawned from the image
	- Can run 10 containers from one image simultaneously
	- In this project, 9 services needs to talk to each other:
		* app -> chroma db address
		* grafana -> prometheus address
		* logstash -> elasticsearch address
	- Layer caching : each instruction (RUN, COPY, FROM) creates a layer.
		* docker caches each layer and only rebuilds from the first changed layer downward
		* dependencies change far less often than the source code
		* COPY source code is last because this changes most often, so it belongs at the bottom
		* EXPOSE is documentation, tells docker which port the container listens on
		* CMD is default command when the container starts
	- Order dockerfile instructions from least-likely-to-change to most-likely-to-change.
		* maximizes cache hits and keeps rebuild times fast
		
		
PHASE 2

* Docker compose => tool that turns 9 separate docker run commands into one
	- when running compose up, all services start, connect to each other and ready
	- networks : connects containers so they can talk to each other by service name
		* bridge driver creates a virtual private network only your containers can see
		* containers find each other by name (chroma, redis, elasticsearch) — no hardcoded IPs
	- volumes : containers are ephemeral. when container stops or is deleted everything written inside it is lost
		* use volume to store data on the host machine, outside the container
		* each service that generates data we need to keep gets its own named volume
			- chroma
			- redis
			- ollama
			- prometheus
			- grafana
			- elasticsearch
	- services : every service follows the same pattern
		* build: . -> docker builds an image from the local dockerfile (only the app service)
		* image: redis:7-alpine pulls a pre builet image from docker hub
		* ports - HOST:CONTAINER, maps a port on your machine to one inside the container
			- 8001:8000 -> chroma db listens on port 8000 inside its container, reach it at localhost:8001 through your browser
		* volumes:
			- named volume (chroma_data:/chroma/chroma), docker managed persistent storage
			- bind mount (./src:/app/src), local folder mounted directly into the container (no rebuild needed, essential for development)
		* environment - env vars injected into the container at startup
			- app service uses env_file: .env to pull secrets in from the .env file
		* depends_on - controls startup order. app service waits for chroma, redis and ollama to start before it launches
		* restart: unless-stopped -> if container crashes, docker restarts it automatically.
		
* docker compose commands
	docker-compose up -d       # start all services in background
	docker-compose down        # stop and remove containers
	docker-compose down -v     # stop + delete all data (full reset)
	docker-compose logs app    # view logs for a specific service
	
* what will happen when running docker compose up -d for the first time?
	- parse the docker compose file and builds dependency graph from all the depends_on to determine startup order
	- pull images from docker hub for every service that uses image:
	- build the app image for the app service which uses build: .
		* docker reads the docker file and builds a custom image layer by layer
	- create volumes, docker creates all 6 named volumes (chroma_data, redis_data, etc) as empty storage buckets on your machine
	- create the network. docker creates the rag-network bridge network that all containers will share
	- start containers in order. Services with no depends_on start first
	- return to the prompt. the -d flag means detached - all containers run in the background and you get your terminal back for work


PHASE 3

* Pydantic settings (Config)
	- read .env for you and maps every variable to the matching field in the class
	- OPENAI_API_KEY in .env becomes settings.openai_api_key in python
	- it validates and coerces types automatically. it points out if there is a string instead of int, etc
	
	* Read the connection signature of each service and create a field for each parameter it requeirs
		- chroma db needs host and port (connect via HTTP)
		- redis needs redis_url (connects via a URL)
		- openai is an api, needs api_key and model name
		- ollama is local HTTP server, needs base_url and model name
		- circuit breaker is pure logic, needs:
			* failure threshold (how many failures before OPEN)
			* recovery timeout (how many seconds before HALF-OPEN probe)
		- RULE: any value that might differ between environments (local dev vs docker vs production) goes into settings
		- Keep all these values in check inside the settings class
		
	* class Config is pydantic way of configuring the settings class itself, metadata about how it behaves

	* @lru_cache => decorator that makes the settings a singleton 
		- caches the return value of a funcion based on its arguments
		- no arguments in get_settings, only one possible cache key - called once
		- it is created the moment any file imports from config.py
		- every other file share the same instance, one sourch of truth for all configuration
		
		
* RAG JSON FORMATTER => json formatter knows how to convert a log record into a json string
	- RAG JSON FORMATTER extends it to inject extra fields into every single log line automatically
	- it will always include the service and level custom fields for every log
	- service field is how to filter and search only this app logs
	
	* why super() and not __init__?
		- add_fields is a method that json formatter calls internally every time it formats a log record
		- not creating new object, rather intercepting the formatting process
		- using super() will let the json formatter do all its normal work
		- once done, add the custom fields (service, level)
		- init would only run once when the formatter object is created.
			* add_fields runs on every log call
			
* Logging => configures the python logging system step by step
	- get logger gets a named logger called "rag"
	- all log calls throught app will use this logger instance
	- log level sets to whatever settings.log_level says (from .env). fallback is logging.INFO (if value is invalid)
	- handlers clear removes any handlers that might already be attached. prevents duplicate log lines
	  if setup_logging is ever called twice
	- stream handler sys.stdout sends log output to stdout, which docker captures and logstash can pick up
	- logger propagate = false, stops the rag logger from passing messages up to the root python logger.
		* without this, every log line would print twice - once from this handler, another from the default root handler
	- no need for lru_cache here. when any file does import logger, python module system makes sure the module is 
	  only executed once, setup_logging only runs once
	  
	  
* PROMETHEUS => application processes requests. without metrics, you have no idea what is happening inside it
	- is it fast or slow? are cache lookups working? is openai failing?
	- prometheus solves this by having app count and measure things as they happen
	- those numbers are stored in memory
	- prometheus visits the /metrics endpoint every 15 seconds and collects them
	- grafana reads from prometheus and draws graphs
	- The flow is:
		Your code increments a counter
			   ↓
		Prometheus scrapes /metrics every 15s
			   ↓
		Grafana reads Prometheus and draws graphs
		
	- three metric types:
		* Counter: a number that only ever goes up. used for
			- request count
			- cache hits
			- llm calls
		  never reset a counter. its wrong if it stops rising
		 
		* Histogram: records individiaul measurements and groups them into buckets. used for:
			- request latency (how long did each request take)
			- chunks retrieved (how many chunks were retrieved per query)
			
		  the buckets [0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0] on request latency means prometheus counts how many
		  requests completed within 0.1s, within 0.25s, etc
		
		* Gauge: a number that can go up or down. Used for current state
			- circuit breaker state (what is the circuit breaker state right now?)
			
		  it is a snapshot of the current value, not a cumulative count
		  
	- what is registry?
		* prometheus client uses a global registry that includes python runtime metrics (memory, cpu, garbage collection)
		* create a custom collector registry so your metrics are isolated - clean and only our app metrics
		* this registry is what gets exported at the /metrics endpoint in app/main.py later
		
	- what are labels?
		* dimensions that let you slice a metric. you have a counter with a layer label:
			- CACHE_HITS.labels(layer="query").inc()
			- CACHE_HITS.labels(layer="embedding").inc()
		* in grafana you can then ask - show me cache hit rate broken down by layer
		
	- IMPORTANT: metrics.py only DEFINES the metrics (declares them into existence)
		* the actual incrementing happens later in the pipeline code:
			- request completes    → REQUEST_COUNT.labels(...).inc()
			- cache returns a hit  → CACHE_HITS.labels(layer="query").inc()
			- circuit breaker changes state → CIRCUIT_BREAKER_STATE.labels(service="chroma").set(1)
		* think of this file as declaring the instruments in an orchestra
		* the data (the music) comes when the instruments are played later in the code
		
		
PHASE 4

* Chunker => implement all three chunking strategies fixed/sentence/recursive

* StrEnum => means the enum members are strings. created chunk strategy as enum with pre-defined choices

* A module-level SEPARATORS dict maps each strategy to its separator list.
  splitter is constructed once
	- noticing the if-elif chain is doing the same thing in every branch but with one different value
	- a dict is almost always the cleaner tool
  
* Dependency injection via default parameter => caller was forced to always construct and pass a config
	- now the function argument config has a sensible default, but the caller can override it
	- dependency injection => the function doesnt control its own configuration, the caller provides it. benefits:
		* testability - pass chunk config in a test without touching the function
		* flexibility - an api endpoint, cli script, batch job can all call the same function with different configs
		* sensible defaults - zero friction for the common case
		
	- WARNING: mutable default arguments (like list = []) are shared across all calls.
	  safe here because ChunkConfig only has primitive fields (int, StrEnum).
	  if the default were a mutable object, each call would share and mutate the same instance:
		def foo(items: list = []):  # BAD - same list object reused across every call
			items.append(1)         # mutates the shared default silently
	  rule of thumb: never use a mutable object (list, dict, set) as a default argument
		
* Dataclass VS pydantic basemodel
	- use dataclass when:
		* data is internal to application (not crossing a boundry like HTTP file or env)
		* trust the caller to pass correct types - no runtime validation needed
		* want something lightweight with zero dependencies
		* the object is simple. just hold values, maybe has a few methods
		
	- use pydantic basemodel when:
		* data comes from an untrusted external source (HTTP request body, json file, cli input)
		* need runtime validation
		* need serialization/deserialization - converting to/from JSON automatically
		* want clear error messages when validation fails
		
	External world (HTTP, .env, files, user input)
			│
			▼
	   [Pydantic]  ← validates and sanitizes here
			│
			▼
	Internal Python code
			│
			▼
	  [@dataclass]  ← just holds data, trust already established
	  
	- if you find yourself adding manual validation to a dataclass with __post_init__
		@dataclass
		class ChunkConfig:
			chunk_size: int = 500

			def __post_init__(self):
				if self.chunk_size <= 0:
					raise ValueError("chunk_size must be positive")
		
		thats a signal you should switch to pydantic.
		
		
		
PHASE 5

* Factory lives in __init__.py => __init__.py is the public API of the package
	- when another module imports the embeddings it should get the embedder
	- shouldnt need to know if its openai or local, an implementation detail
	- Centeralizing it in __init__.py means changing the logic in exactly one place
	- __init__.py is the front desk, it decidies which back room to use and hands you the result
	- callers only talk to the front desk
	
* lru_cache => caches the return value of a function based on its arguments
	- no arguments, only one possible cache key. makes each embedder effectively a singelton
	- loading all-minilm-l6-v2 downloads the model weights and load them into memory, an expensive operation
	- without lru cache, every call to get_local_embeddgins function would reload the model from disk
	- maxsize=1 is explicit intent, i only want to cache one result.
	- default maxsize is 128 which implies caching many different inputs, less readable for the embedding use case
	
	with lru_cache:
		First call  → loads model weights into RAM (~seconds)
		Second call → returns the exact same object instantly (cached)
		Third call  → same cached object
		
* OpenAI vs Local embedder parameters => more control = more parameters
	- OpenAI runs on their servers, you just send text over HTTP and get a vector back
		* only needs: model name + api key to authenticate. everything else is their problem
	- Local runs inside your process, you own the hardware and encoding decisions:
		* model_name - which model to download from HuggingFace Hub
		* model_kwargs: device - cpu or cuda (gpu). a hardware decision you now own
		* encode_kwargs: normalize_embeddings=True - scales every output vector to length 1
		  required for cosine similarity to work correctly. without it, vectors close in direction
		  but different in magnitude would score incorrectly


PHASE 6

* Redis Cache => three-layer cache (query, embedding, llm). each layer is a key-value store with a TTL

* make_key => generates a Redis key using SHA-256 hash of the content
	- SHA-256 is deterministic: same input always produces same output, regardless of who calls it
	- two users asking "What is RAG?" produce the exact same hash => same Redis key => cache hit
	- hexdigest() converts raw bytes into a hex string (0-9, a-f only) - always 64 characters, always safe
	- key format: rag:layer:digest (e.g. rag:query:4c1f3b2a...)
	  the prefix is a Redis namespace convention - lets you scan/filter by layer

* safe_get / safe_set => graceful degradation for Redis failures
	- Redis is an optional optimization, not a core requirement
	- without try/except: Redis crash -> exception propagates -> user gets a 500 error
	- with safe_get/safe_set: Redis crash -> warning logged -> returns None -> pipeline continues without cache
	- setex = set + expiry in one atomic operation (key, ttl, value)

* Cache-Aside Pattern (Lazy Loading) => the standard workflow for any cache:
	1. generate cache key from input
	2. check Redis (GET)
	3. HIT  - return cached result immediately
	4. MISS - do expensive work (LLM call / embedding / vector search)
	5.      - store result in Redis (SET with TTL)
	6.      - return result to caller

	three rules that always apply:
	- always check cache first
	- always set after a miss (so the next caller benefits)
	- always set a TTL (so stale data doesnt live forever)

* metrics are updated inside get methods only (not set)
	- a hit/miss only makes sense on a read. writing is always a write, nothing to measure
	- cache owns its own observability - pipeline just calls get_query() without caring about metrics
	
	
PHASE 7

* circuit breaker state machine => this defines the data structure and transition logic for one service health status
	- we define the instruments, in this case the service name, state. 
	- later the pipeline plays them when needed
	
* One breaker per external service => one circuit breaker per independently failable external dependency
	- if chroma db is down, you dont want to stop calling openai. they are unrealted failures
	- each is assigned its own breaker so failures are isolated
	- each breaker is a singelton, imported by pipeline code and shared across all requests
	
* @property in a dataclass => use when a value needs to be computed or evaluated at the moment it's read
	- plain field: value stored, static until explicitly changed
	- @property: computed fresh on every read, can run logic on access, you control get/set separately
	
	three scenarios:
	1. derived values - computed from other fields, no need to store separately
	   (e.g. word_count = len(content.split()) - storing it would require keeping it in sync manually)
	2. side-effect on read - reading triggers a state transition or metric update (circuit breaker state)
	3. controlled access - field is readable but not settable from outside (no setter defined = read-only)
	
	rule: if the value depends on something that changes over time (like time.time()) or is derived
	      from other fields -> make it a property. if its just data being held -> plain field.

* state property is a self-advancing state machine
	- every time anything reads the state, it silently checks if the timeout expired
	- auto-transitions OPEN -> HALF_OPEN without any external trigger
	- is_available() returns True for CLOSED and HALF_OPEN (probe must get through), False for OPEN (fail fast)
	
	- two names, intentional:
		* _state  => actual stored value, private, set directly with self._state = ... (no property, no recursion)
		* state   => public interface, @property, triggers timeout check on every read, returns _state
	- accessing self.state runs the full property body each time
	  if we returned self._state directly without the check, the circuit would stay OPEN forever

* why dataclass and not pydantic here?
	- pydantic validates data ONCE at construction time. its for data shapes at trust boundaries
	- circuit breaker manages evolving runtime state - failure_count increments, state transitions over time
	  none of that is input validation, its behavioral logic responding to live events
	- pydantic models are immutable by default - fighting the framework to mutate _state and _failure_count
	
	sharpened rule:
	| Use Pydantic               | Use dataclass                          |
	| validate data at boundary  | manage state that evolves over time    |
	| data crosses trust boundary| data is internal and trusted           |
	| need serialization         | need methods with side effects         |


PHASE 8

* BM25 fallback retriever => activated when ChromaDB is unavailable (circuit breaker OPEN)
	- instead of returning an error, fall back to classic keyword matching
	- system keeps running, semantic understanding is lost but results still returned
	- warning log fires in __init__ the moment the fallback is instantiated

* corpus => NLP term for "the full collection of text you are working with"
	- BM25Okapi expects pre-tokenized lists, not raw strings
	- corpus = [doc.page_content.lower().split() for doc in docs]
	  ["What is RAG?"] -> [["what", "is", "rag?"]]

* tokens => the query in the same tokenized form as the corpus
	- BM25 compares query tokens against document tokens, not raw strings
	- tokens = query.lower().split()

* get_scores => scores every document in the corpus against the query tokens
	- returns a list of floats, one per document. higher = more relevant
	- documents with no matching tokens score 0

* top_indices => sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:k]
	- range(len(scores)) generates indices [0, 1, 2, ...], one per document
	- sorted by score value, reverse=True means highest score first
	- [:k] takes only the top k indices
	- why sort indices and not scores? need the index to look up the actual document
	  sorting scores directly loses track of which score belongs to which document
	- results = [self.docs[i] for i in top_indices] — pulls documents in ranked order
	
	
PHASE 9

* Abstract base class (ABC) => python way of enforcing the contract
	- the rest of the system needs to call an LLM
	- which LLM should be swappable at runtime? openai when healthy, ollama as fallback
	- without an interface the chain would have to know about both
	- the LLMs are interchangeable because both are guaranteed to have get_llm and provider_name, because of the contract
	- @abstractmethod enforcement: if a subclass forgets to implement get_llm(), Python raises TypeError
	  at instantiation time - before any code runs. compile-time-style safety in a dynamic language

* When interface is the right tool?
	- i have multiple things that need to be interchangeable
	- multiple implementations of the same concept. openai and ollama are both an llm you can call
	- the caller shouldnt need to know which one it got, the chain just wants an LLM
	- Liskov Substitution Principle (the L in SOLID): any subclass can stand in for the parent without breaking callers
	- the implementation will change at runtime or per environment
		* local dev uses ollama
		* production uses openai
		* fallback switches automatically
	- you want to enforce a contract across a team/codebase
		* when adding anthropic provider, abc forces them to implement get_llm to make it work
	- the interface is great when substitutability is the goal
		
* When interface is not the right tool
	- there is only ever going to be one implementation and no swap is needed
	- abc adds complexity with no benefit, just use the class directly


PHASE 10

* indexer.py => bridge between raw files and the vector database
	- load_knowledge_base: walks the folder, loads .md files as Document objects
	- ingest_documents: chunk -> embed -> store in one pipeline (3 lines)
		chunks = chunk_documents(docs, config)   # split into small pieces
		vectorstore = get_vectorstore()          # connect to ChromaDB
		vectorstore.add_documents(chunks)        # embed each chunk and store

* DirectoryLoader + TextLoader
	- DirectoryLoader walks a folder and loads all files matching the glob pattern
	- glob="**/*.md" => ** means any subdirectory at any depth, *.md means any markdown file
	  "*.md" (no **) would only find files in the root folder, not subfolders
	- loader_cls=TextLoader => reads each file as raw text, wraps it in a Document with file path as metadata
	- output: one Document per .md file, containing full content + path metadata

* Chroma and collection name
	- chromadb.HttpClient connects to ChromaDB running in Docker as an independent HTTP server
	- collection_name => ChromaDB organizes vectors into collections, like tables in SQL
	  one server can hold many collections. specifying a name keeps this project isolated
	  without it you risk mixing data from different projects
	- embedding_function => must use the same embedding model for both storing and searching
	  if you indexed with OpenAI but searched with local model, vectors would be in different
	  spaces and similarity scores would be meaningless

* add_documents => does two things in one call
	- embeds each chunk using the embedding function provided
	- stores (chunk text, vector, metadata) together in ChromaDB
	- this is the moment .md files become searchable by meaning

* functions vs class => use a class when you need to hold state between calls
	- RAGRedisCache holds _client, CircuitBreaker holds _failure_count/_state - they need a class
	- indexer functions are stateless: data goes in, data comes out, nothing persists between calls
	- wrapping stateless functions in a class adds self and instantiation with no benefit
	
	| Use a class                              | Use functions                        |
	| need to hold state between calls         | each call is self-contained          |
	| need multiple instances with diff config | one shared behavior, no variation    |
	| object has identity or lifecycle         | pure transformation of input/output  |
	
	- exception: if indexer needed a persistent vectorstore connection to avoid reconnecting on every
	  call, a class would make sense. until that optimization is needed, functions keep it simpler.


PHASE 11

* retriever.py => orchestrates primary (ChromaDB) and fallback (BM25) retrieval
	- chroma available: similarity_search -> record_success -> return chunks
	- chroma fails: record_failure -> fall back to BM25 -> return chunks
	- no retriever available: log error, return empty list

* _bm25_fallback is a module-level global (pre-loaded singleton)
	- BM25FallbackRetriever holds state (bm25 index + full docs corpus) so it IS a class
	- but the instance lives at module level, created ONCE at startup via set_bm25_fallback(docs)
	- reason: BM25 needs the full corpus loaded into memory upfront, before the first failure
	  you cant build it at query time when ChromaDB just failed - too late, no time
	- underscore prefix (_bm25_fallback) signals it is private to this module

* retrieve() is a function, not a class
	- stateless coordination between two external things: ChromaDB call and BM25 singleton
	- no state to hold between calls, functions are the right fit
	
* the pattern: stateful object, stored as module-level singleton, accessed by stateless functions
	- you have seen this before: settings, logger, cache, circuit breakers - same pattern

* record_success / record_failure => for the circuit breaker, not prometheus directly
	- updates the circuit breaker state machine (chroma_cb) - counts failures to decide when to open
	- prometheus does get updated as a side effect via _update_metric() inside those calls
	- but the primary purpose is resilience logic, not metrics

* CHUNKS_RETRIEVED.observe(len(docs)) => histogram measurement
	- .observe(value) records one data point. each query call adds one measurement
	- over time prometheus accumulates all observations to answer:
	  "what is the average number of chunks retrieved?" or "how many queries returned fewer than 3 chunks?"
	- this tells you about retrieval quality over time

* metric method summary
	- .inc()          Counter    increment by 1
	- .set(value)     Gauge      set to this exact value right now
	- .observe(value) Histogram  record one measurement of this value
	
	
* Generator => the augmentation part of RAG
	- get the selected documents and turn them into single string
	- each document separated by its source file and new lines
	- the entire string become the context inside RAG PROMPT
	- LLM sees the chunks with their labels so it knows which module to cite
	
	
* LCEL chain => the | is langchain operator overloaded on a base class called Runnable
	- every component in langchain, prompt | llm | parser, extends Runnable
	- the | just chains them - output of each feeds as te input to the next
	- RAG_PROMPT.invoke({"context": ..., "question": ...})
		→ formatted prompt messages
			→ llm.invoke(messages)
				→ raw LLM response object
					→ StrOutputParser.invoke(response)
						→ plain string
						
						
* chain.invoke({"context": context, "question": question})
	- not kwargs
	- single positional argument, a plain python dict
	- langcain matches the keys context and question to the context and question placeholders in the prompt template
	- standard pattern across all LCEL chains
	
	
* Raise => raise with no argument re-raises the same exception up the call stack
	- intentional, generator does not decide what the user sees
	- generator.py  →  raises exception
			↓
		chain.py      →  catches it, logs it, decides what to return to the API
			↓
		api/routes/chat.py  →  returns an HTTP response to the user
	- if returned a string like error: something... the caller (chain) would have no way to know that generation failed
	- by re-raising, it lets the layer above decide - retry with Ollama? return a 500? show friendly message?


* Let it propagate => errors bubble up until they reach a layer that has enough context to make a decision
	- raise when you can detect a failure but cannot meaningfully recover from it at your level
	- recover locally if you can, raise if you cannot
	- each layer has a different job:
		generator.py       => I only know HOW to generate. I don't know what the user should see.
			↓ raises
		chain.py           => I know the full pipeline. I can retry, fallback, or give up gracefully.
			↓ raises (or returns a safe response)
		api/routes/chat.py => I know about HTTP. I decide the status code and response body.
			↓
		User sees a message

	- this principle is applied consistently across the codebase:
		* redis_cache.py       => swallows errors. a cache failure is recoverable right there - return None and continue.
		                          no layer above needs to know
		* circuit_breaker.py   => records the failure, flips state. does not raise
		* generator.py         => raises. only chain.py knows whether to retry with Ollama or give up


PHASE 11

* SessionMemory => in-memory conversation buffer per browser tab
	- Message dataclass holds one chat turn (role + content)
	- use dataclass and not pydantic because:
		* message is internal data, never crosses a trust boundary (no HTTP, no file, no env)
		* no runtime validation needed - role is always set by our own code, never by untrusted input
		* needs mutable state - the list grows every turn
	- SessionMemory holds a dict of session_id => list of Messages

* session_id => a UUID generated per browser tab by the UI, not by the user
	- generated once in ui.py when the tab opens: str(uuid.uuid4())
	- scoped to a browser tab - two tabs = two independent conversation histories
	- ephemeral - lives only in server memory. restart the server and it's gone
	- passed down through every layer as a plain string:
		ui.py (generates UUID)
			↓
		api/routes/chat.py (receives it in ChatRequest body)
			↓
		chain.py (passes it to session_memory calls)
			↓
		SessionMemory (uses it as the dict key)
	- since it is just a string, replacing it with a JWT or auth token later requires
	  zero changes to SessionMemory - the memory layer doesn't care what the string is

* defaultdict(list) => defaultdict takes a callable as its argument
	- when a key doesn't exist yet, it calls that callable to produce the default value
	- list is the type itself (calling list() returns [])
	- so on first access for any session_id, it automatically creates an empty list []
	- no need to check "if session_id not in self._sessions" before appending

	normal dict vs defaultdict:
		d = {}
		d["abc"].append("x")            # KeyError: 'abc'

		d = defaultdict(list)
		d["abc"].append("x")            # works - creates [] first, then appends

* format_history => only injects the last 10 messages into the prompt
	- full history is stored in memory (unbounded)
	- only last 10 turns are injected - injecting the full history would blow up the context window


* Python module singleton => Python handles this automatically via sys.modules
	- when a module is imported for the first time:
		1. Python executes the module top to bottom
		2. creates any module-level instances (e.g. session_memory = SessionMemory())
		3. stores the result in sys.modules (a dict of module name => module object)
	- every subsequent import of the same module skips execution entirely
	  and returns the cached version from sys.modules
	- the same instance is shared across every file that imports it

	First import  => execute module => create instance => store in sys.modules
	Second import => find module in sys.modules => return it => same instance

	- this is why the singleton pattern in Python is simpler than other languages:
		* Java / C++ => need explicit singleton logic: private constructor, static instance, thread locks
		* Python     => just write a module-level variable, the import system handles the rest

	- examples in this codebase, all singletons via module-level variable:
		* settings        => app/core/config.py
		* logger          => app/core/logging_config.py
		* cache           => rag/cache/redis_cache.py
		* chroma_cb / openai_cb / redis_cb  => rag/resilience/circuit_breaker.py
		* session_memory  => rag/memory/conversation.py

	- lru_cache on get_settings() is technically redundant (module only runs once anyway)
	  it is there as explicit intent - a signal to the reader that this is meant to be a singleton


* Open/Closed Principle (the O in SOLID) => open for extension, closed for modification from the outside
	- a module's internal implementation can change freely
	- but its interface (function signatures, inputs, outputs) stays stable
	- callers never need to change when the internals swap

	profiles.py example:
		today  => load_profile() reads from a JSON file on disk
		later  => load_profile() queries PostgreSQL
		result => every caller stays untouched. same input, same output, different internals

	- this pattern appears in three places in this codebase:
		* profiles.py          => JSON today, PostgreSQL later
		* providers/__init__.py => OpenAI today, Ollama as fallback
		* embeddings/__init__.py => local or OpenAI based on env var

	- the key insight: hide the HOW behind a stable interface
	  callers only know WHAT the function does, not how it does it


* Design patterns in this codebase => three distinct patterns, each solving a different problem

	Facade => hides a complex subsystem behind a simple call
		- the complexity is coordinating multiple components
		- example: run_rag_pipeline() in chain.py
		  one call that hides: retrieve → cache check → generate → store memory → update metrics
		- caller just calls the function and gets an answer back

	Open/Closed => interface stays stable, implementation can swap without breaking callers
		- not necessarily about hiding complexity, about replaceability
		- example: load_profile() — today reads JSON, later queries PostgreSQL
		  zero changes needed in callers when the storage mechanism changes

	Strategy => multiple implementations of the same interface, selected at runtime
		- caller doesn't know or care which implementation runs
		- examples in this codebase:
			* get_provider()    => picks OpenAI or Ollama based on circuit breaker state
			* get_embeddings()  => picks local HuggingFace or OpenAI based on env var
			* load_profile()    => will pick JSON or PostgreSQL

	one-line distinction:
		Facade   => simplify a complex workflow
		Open/Closed => make implementations swappable without breaking callers
		Strategy => pick between multiple implementations at runtime


* conversation.py vs profiles.py => separation of concerns in the memory layer
	- two files, two different questions, two different storage strategies

	| What                        | Where             | Storage    | Survives restart |
	| what the user said          | conversation.py   | in-memory  | No               |
	| who the user is             | profiles.py       | JSON file  | Yes              |

	- conversation.py => "what did we talk about this session?"
	  stores the actual messages (human + assistant turns) per session_id
	  ephemeral - lives only in server memory

	- profiles.py => "have I seen this user before, and how often do they visit?"
	  stores identity metadata only (first_seen, last_seen, query_count)
	  does NOT store message content - that is not its responsibility

	- if you wanted to persist conversation history across restarts in the future:
	  that would be a separate addition (saving messages to PostgreSQL or a file)
	  profiles.py would not own that - it answers a different question


PHASE 12

* chain.py => the heart of the system. the facade that wires every component into one pipeline
	- single function: run_rag_pipeline(question, session_id, user_id) -> dict
	- caller asks a question, gets back an answer + observability metadata
	- the backbone of every step: check if the result exists in cache, if not create it

* Setup
	- time.perf_counter() starts a high-precision timer for latency measurement
	- trace_id is the first 8 chars of a UUID - used to correlate all log lines from this one request
	  search logs by trace_id to find every log line that belongs to this request
	- session_id falls back to "default" if nothing was passed

* Step 1 - Query cache check
	- check if this exact question was already answered before
	- if yes: calculate latency and return immediately
	  retrieval and generation are completely skipped - fastest possible path
	- notice chunks: [] in this early return - we have no chunks because we never retrieved any
	- cache_hit: "query" tells the UI to show the cache badge

* Step 2 - Retrieve relevant chunks
	- calls retrieve(question, k=4)
	- retrieve() handles ChromaDB -> BM25 fallback transparently
	- chain.py doesn't know or care which retriever ran

* Step 3 - LLM cache check
	- prompt_key = question + first 100 chars of each retrieved chunk
	- why not cache by question alone here?
	  the same question could return different chunks if the knowledge base changes
	  this key captures both the question AND the context - a fingerprint of the exact prompt
	- if hit: use cached answer, mark cache_hit = "llm"
	- if miss: fall through to Step 4

* Step 4 - Generate (only runs on LLM cache miss)
	- calls generate(question, docs) which runs the LCEL chain
	- immediately stores the result in LLM cache so the next identical call skips this step

* Step 5 - Store everything
	- stores the final answer in query cache (so Step 1 hits on the next identical question)
	- adds the question and answer to session memory
	- updates user profile if user_id was provided

* Return dict => the answer plus observability metadata
	{
	    "answer":     the actual text the user sees
	    "cache_hit":  "query" | "llm" | "none" — which cache layer saved work
	    "chunks":     the retrieved docs (first 200 chars + source path each)
	    "latency_ms": total time from pipeline start to finish in milliseconds
	    "trace_id":   short ID to find all log lines for this exact request in Kibana
	}
	- the UI uses all of it: shows the answer, renders the cache badge, displays latency,
	  and shows the retrieved chunks in a collapsible panel

* Cache layers and what each one saves
	query cache hit  => skip retrieval + generation entirely (fastest path)
	llm cache hit    => skip generation only, retrieval still runs
	no cache hit     => full pipeline: retrieve -> generate -> store
	
	
	
PHASE 13

* With, the context manager => python way of running setup before a block, and perform teardown after it
	- guaranteed, even if an exception occurs. it will be closed 100%
	- the with block will handle the close operation himself
	- enter → your code runs → exit (no matter what)
	
* REQUEST_LATENCY.labels(endpoint="/chat").time() => prometheus helper that:
	- on enter (with starts): records time.time(), notes when the block began
	- on exit (with ends): calculates now - start and calls .observe() on the histogram
	- wrapped in with is clear and guaranteed that if run rag pipeline raises an excpetion, the latency
	  is still recorded.
	- run rag pipeline is the entire cost of this endpoint. wrapping it gives prometheus an accurate measurement 
	  of total request duration as seen from the API layer
	- run rag pipeline tracks its own latency ms internally for the user to see
	- prometheus here is for grafana dashboards
	- two different consumers, same underlying work being measured


* documents.py => ingest endpoint. only input is a file, no request schema needed
	- FastAPI's UploadFile handles the file type automatically, no Pydantic schema required
	- schemas (like ChatRequest) are for structured JSON bodies. a file upload is a different content type

* returning a plain dict in FastAPI => no need for JsonResponse
	- when a route returns a plain dict, FastAPI automatically serializes it to JSON
	  and sends it with Content-Type: application/json
	- JsonResponse is only needed for manual control: custom status codes, custom headers,
	  or pre-serialized JSON strings
	- returning a dict is the standard FastAPI idiom

* await file.read() => endpoint is async, file.read() is an async operation
	- await pauses this coroutine until the file bytes arrive, without blocking other requests
	- destination.write_bytes() then writes the raw bytes to disk synchronously (small files, acceptable)

* where uploaded files are stored
	- data/uploads/ is a path on the host machine — files written there survive server restarts
	- in Docker, data/ is a bind mount: ./data:/app/data in docker-compose.yml
	  files written inside the container land directly in the host's data/ folder
	- survive: container restarts, rebuilds, docker-compose down
	- only manually deleting the folder would lose them (docker-compose down -v does NOT touch bind mounts)

* two separate stores for uploaded content
	- data/uploads/     => the original raw file (permanent, on disk, bind mount)
	- ChromaDB          => the embedded chunks (in Docker named volume chroma_data)
	- if ChromaDB is wiped but data/uploads/ is intact, you can re-ingest
	- if both are lost, the content is gone


* Health endpoints => two endpoints, two different audiences, two different purposes

	/api/health => liveness probe. for infrastructure (Docker, Kubernetes, load balancers)
		- answers one question: "is the process alive and able to handle requests?"
		- intentionally does nothing - no DB check, no Redis ping
		- if the process can respond at all, it returns {"status": "ok"}
		- that is enough. infrastructure doesn't care about internal service health
		- wired in docker-compose as:
			healthcheck:
			  test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]
			  interval: 30s
			  retries: 3
		- if it stops responding, Docker marks the container unhealthy and can restart it

	/api/health/circuit-breakers => operational visibility. for developers and operators
		- answers: "which internal services are degraded right now?"
		- returns live state of all three circuit breakers (chroma, openai, redis)
		- example response:
			{
			  "circuit_breakers": [
			    {"service": "chroma",  "state": "OPEN",   "failure_count": 5},
			    {"service": "openai",  "state": "CLOSED", "failure_count": 0},
			    {"service": "redis",   "state": "CLOSED", "failure_count": 0}
			  ]
			}
		- typical use: user reports slow or wrong answers
		  hit this endpoint, see state: OPEN for chroma
		  now you know ChromaDB is down and BM25 fallback is running instead of semantic search
		- Prometheus already tracks CIRCUIT_BREAKER_STATE as a gauge for Grafana dashboards
		  this endpoint gives a human-readable instant snapshot without opening Grafana

	one-line distinction:
		/health                   => "is the app running?"       → for infrastructure
		/health/circuit-breakers  => "what is degraded right now?" → for humans

* Why /health does NOT check dependencies (Redis, ChromaDB, OpenAI)
	- a liveness probe should only fail if the app itself is broken
	- if Redis is down but the app is running, the app should still serve requests (cache degrades gracefully)
	- if the liveness probe checked Redis and Redis was down, Docker would restart the app
	  restarting the app does not fix Redis — it just creates unnecessary churn and downtime
	- infrastructure probes and service health are separate concerns:
		* liveness probe  => is this container healthy? should it be restarted?
		* circuit breaker => is this dependency healthy? should requests be rerouted?


* Liveness vs Readiness => two distinct probe types, each with a different consequence when they fail

	| Probe      | Question                              | Fails →                                        |
	| Liveness   | "Is the process alive?"               | Container is RESTARTED                         |
	| Readiness  | "Is the app ready to accept traffic?" | Container is REMOVED from load balancer        |

	- liveness failure => something is fundamentally wrong with the process. restart it.
	- readiness failure => dependencies aren't ready yet. stop sending traffic, but don't restart.
	  useful at startup: ChromaDB may still be initializing. don't serve users until it's up.

* /api/health/ready => readiness probe implementation
	- pings Redis using cache._get_client().ping()
	- pings ChromaDB using get_vectorstore()
	- builds a checks dict: {"redis": "ok", "chroma": "ok"} or "unavailable" per service
	- returns JSONResponse (not plain dict) because status code must be controlled:
		* 200 if all services are ok   → load balancer keeps routing traffic here
		* 503 if any service is down   → load balancer stops sending traffic to this instance
	- a plain dict return always gives 200, which would mislead the load balancer
	  into thinking the app is ready when it cannot serve correctly

	example responses:
		all healthy:  {"status": {"redis": "ok", "chroma": "ok"}}           HTTP 200
		chroma down:  {"status": {"redis": "ok", "chroma": "unavailable"}}  HTTP 503

* three health endpoints, three different audiences
	/api/health          → infrastructure (Docker, Kubernetes): is the process alive?
	/api/health/ready    → load balancer: are dependencies ready to serve traffic?
	/api/health/circuit-breakers → developers: which services are currently degraded?


* Docker healthcheck block => tells Docker how to test if a container is healthy
	healthcheck:
	  test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]
	  interval: 30s       # how often to run the check
	  timeout: 10s        # how long to wait for a response before marking as failed
	  retries: 3          # how many consecutive failures before marking container unhealthy
	  start_period: 40s   # grace period at startup - failures during this window don't count

	- start_period is critical for the app container
	  the app needs time to boot, load the knowledge base, and connect to ChromaDB
	  without start_period, Docker would count the first 3 slow startup probes as failures
	  and mark the container unhealthy before it even finishes starting

	- test command runs INSIDE the container, not on the host machine
	  this means the tool used (curl, redis-cli) must be installed inside the container image:
		* app container    => curl installed via apt-get in Dockerfile
		* redis container  => redis-cli is built into the redis:7-alpine image
		* chroma container => curl available in the chromadb image

* depends_on: list format vs map format in Docker Compose
	- two valid syntaxes, cannot be mixed

	list format (simple, no conditions):
		depends_on:
		  - chroma
		  - redis

	map format (required when using condition):
		depends_on:
		  chroma:
		    condition: service_healthy
		  redis:
		    condition: service_healthy

	- condition: service_healthy => waits until the service's healthcheck passes
	- condition: service_started => waits only for the container to start (no healthcheck needed)
	- use service_started for services without a healthcheck (e.g. ollama)
	- common mistake: trying to add condition as a child of a list item
	  "- chroma" followed by indented "condition:" is invalid YAML
	  the entire depends_on block must use map format if any service uses condition


PHASE 14

* main.py => the entry point and wiring layer. deliberately thin - owns no business logic
	- routes live in api/routes/
	- pipeline lives in rag/
	- config lives in app/core/
	- main.py only connects them together

* lifespan => startup and shutdown logic for the FastAPI app
	- decorated with @asynccontextmanager because FastAPI is async and startup can involve
	  async operations (network calls to ChromaDB)
	- everything BEFORE yield runs at startup, before the first request is handled
	- yield is the pause point: "now run the application and serve requests"
	- everything AFTER yield runs at shutdown (closing connections, flushing queues, etc.)

	server starts → before yield → yield → handles requests → after yield → server stops

	what happens at startup in this project:
		1. load the knowledge base markdown files into memory
		2. set BM25 fallback with those docs (must be loaded upfront, before any failure occurs)
		3. check if ChromaDB collection is empty → if yes, ingest and embed everything
		4. log "RAG system ready"

* startup validation and graceful startup failure
	- ChromaDB check runs at startup so the first user request never triggers a slow ingest
	  the system is ready before traffic arrives
	- the ChromaDB check is wrapped in try/except
	  if ChromaDB is unavailable at startup, the app still starts
	  BM25 fallback is already loaded, warning is logged, server comes up anyway
	- this is graceful degradation applied at startup, not just at request time

* creating the FastAPI app
	app = FastAPI(title="...", description="...", version="1.0.0", lifespan=lifespan)
	- FastAPI() creates the application instance
	- lifespan= wires in the startup/shutdown function
	- title, description, version appear in the auto-generated /docs (Swagger UI) page

* middleware => wraps every single HTTP request automatically
	- runs before the route handler and after it
	  Request arrives → middleware enter → route handler runs → middleware exit → response sent
	- call_next(request) is the call that actually runs the matching route handler
	- why use middleware instead of logging inside each route?
	  it is cross-cutting - every endpoint gets the same behavior automatically
	  one place, universal coverage. you can never forget to add it to a new route
	- this middleware logs method, path, status code, and latency for every request
	  feeds into Logstash → Elasticsearch → Kibana

* /metrics endpoint => exposes Prometheus data for Grafana
	- generate_latest(REGISTRY) is a prometheus_client function that serializes all metrics
	  in the registry into Prometheus text format (plain text, not JSON)
	- REGISTRY is the custom isolated collector from metrics.py - only app metrics, no Python runtime noise
	- Prometheus scrapes this endpoint every 15s (configured in prometheus.yml)
	- Grafana reads from Prometheus and draws graphs
	- Response (not plain dict) because content is already a formatted string and needs
	  media_type=CONTENT_TYPE_LATEST so Prometheus knows how to parse it

* registering routes => routes in separate files don't automatically become part of the app
	- must explicitly register each router:
		app.include_router(chat.router)
		app.include_router(documents.router)
		app.include_router(health.router)
	- each router was created with APIRouter(prefix="/api", ...) in its own file
	- include_router mounts all routes from that router onto the main app
	- without this, the endpoints simply don't exist

* NiceGUI mounted last => setup_ui(app) is always the final line
	- NiceGUI's ui.run_with() takes over the root path /
	- must be registered after all API routes
	- if it were first, it could intercept API calls before they reach their handlers

PHASE 16

* two separate observability pipelines
	METRICS pipeline:  app → Prometheus → Grafana
	LOGS pipeline:     app → Logstash → Elasticsearch → Kibana

	they answer different questions:
		metrics => numbers over time (how fast? how many? what rate?)
		logs    => what exactly happened, when, and with what context

* METRICS pipeline in detail
	app/core/metrics.py     defines the counters, histograms, gauges
	/metrics endpoint       serializes them as Prometheus text format on demand
	prometheus.yml          tells Prometheus to scrape app:8000/metrics every 15s
	Prometheus              stores the time-series numbers
	Grafana ds.yml          auto-provisions Prometheus as the default datasource on startup
	Grafana dash.yml        tells Grafana where to look for dashboard JSON files
	rag_dashboard.json      pre-built panels: p95 latency, cache hit rate, LLM calls, circuit breaker states

* LOGS pipeline in detail
	app/core/logging_config.py   formats every log line as JSON (not plain text)
	stdout                       Docker captures stdout from the container
	logstash.conf                Logstash listens on TCP 5044, receives JSON lines
	                             filters: if service == "rag-from-scratch", tags the log
	                             output: forwards to Elasticsearch with daily index (rag-logs-YYYY.MM.dd)
	Elasticsearch                stores and indexes the structured log documents
	Kibana                       UI to search, filter, and visualize logs by field (trace_id, session_id, latency_ms)

	prometheus/prometheus.yml           => scrape config: which app to pull metrics from, how often
	grafana/provisioning/datasources/   => auto-connect Grafana to Prometheus on first startup (no manual setup)
	grafana/provisioning/dashboards/    => tell Grafana which folder to scan for dashboard JSON files
	grafana/dashboards/rag_dashboard.json => the actual dashboard panels with PromQL queries
	logstash/logstash.conf              => log pipeline: input (TCP) → filter (tag by service) → output (Elasticsearch)

* why JSON logs matter
	- plain text logs: "Request completed in 142ms" — hard to filter, impossible to query by field
	- JSON logs: {"latency_ms": 142, "trace_id": "a3f8c1", "session_id": "b7d2"} 
	  Kibana can filter by any field instantly
	  find all requests slower than 500ms: latency_ms > 500
	  find all logs for one request: trace_id: "a3f8c1"
	- this is why RAGJsonFormatter in logging_config.py adds service and level to every log line


DOCKER-FIRST-RUN TROUBLESHOOTING (war journal)

* this section documents real failures hit while bringing rag-from-scratch up on Docker Desktop

* Chroma healthcheck => chromadb/chroma image is minimal at first glance
	- /usr/local/bin may only show the chroma binary — curl, wget, python often NOT on PATH for healthcheck CMD
	- documented curl http://localhost:8000/api/v1/heartbeat fails: wrong path for current Chroma, or curl missing
	- current Chroma heartbeat path is /api/v2/heartbeat (check official docs if image upgrades)
	- working approach without curl: TCP-only probe inside the container (bash + /dev/tcp), e.g.
		timeout 2 bash -c 'echo > /dev/tcp/127.0.0.1/8000'
	  proves the server is listening — weaker than HTTP 200 but enough for depends_on: service_healthy
	- add start_period: 60s on Chroma so slow boots do not flip unhealthy before the server binds
	- alternative: remove Chroma healthcheck and use depends_on condition: service_started for app
	  rely on /api/health/ready on the FastAPI app for real dependency checks

* depends_on map syntax => when using condition: service_healthy, the whole block must be map form:
		chroma:
		  condition: service_healthy
	  not a list with "- chroma" plus inline condition (invalid YAML)

* ollama has no healthcheck => use condition: service_started, never service_healthy
	  otherwise app waits forever for a health signal that never arrives

* uvicorn: executable file not found in $PATH
	- uv sync installs CLI tools into /app/.venv/bin — that path is not on PATH by default
	- fix: CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
	- after changing Dockerfile CMD, must rebuild the image: docker compose build --no-cache app

* Pydantic Settings + .env => extra_forbidden / ValidationError on prometheus_port (or any var)
	- BaseSettings in Pydantic v2 rejects ANY environment variable that does not map to a declared field,
	  unless you allow extras

PHASE 19

* Auth configuration surface => JWT and SQLite live in environment contract + Settings
	- .env.example documents jwt_secret, jwt_algorithm, access_token_expire_minutes, sqlite_db_path,
	  allow_annonymous_chat (and related vars as the plan adds them)
	- app/core/config.py maps those names to typed fields on Settings (pydantic-settings reads .env automatically)
	- sqlite_db_path default: "data/app_users.db" — relative paths resolve against the process cwd unless overridden with an absolute path

* src/app/auth/ => persistence helpers for registered users (not HTTP). app/auth/db.py owns connection + DDL + CRUD helpers for the users table

* SQLite and SQL => SQLite executes standard SQL DDL/DML (CREATE TABLE, INSERT, SELECT, etc.) with SQLite-specific details (dynamic typing, TEXT for ISO timestamps). The auth module uses plain parameterized SQL strings executed via sqlite3.Connection.execute()

* Patterns in app/auth/db.py
	- CREATE TABLE: run once from init_user_db(), use CREATE TABLE IF NOT EXISTS with columns, types, and constraints (PRIMARY KEY, NOT NULL, UNIQUE). Follow with conn.commit() so schema changes persist
	- INSERT: list explicit columns, use VALUES (?, ?, ...) and pass a tuple of values as the second argument to execute()
	- SELECT by filter: SELECT ... FROM users WHERE column = ?, tuple binds one value per placeholder. fetchone() returns one row or None

* Parameter placeholders (?) and SQL injection => never concatenate user input into the SQL text. Placeholders keep values separate from parsing; SQLite binds them safely. Injection risk returns only if raw strings are interpolated into the query

* commit() => ends the current transaction and makes pending writes durable in the database file on disk. Without commit(), implicit transactions may roll back on close depending on context — the auth helpers commit after DDL and after INSERT

* Where the .db file comes from => sqlite3.connect(path) opens the file; if it does not exist yet, SQLite creates it at that path. _connect() ensures path.parent.mkdir(parents=True, exist_ok=True) so the directory exists first. First real writes (CREATE TABLE / INSERT) populate structure and data

* dict(row) => with conn.row_factory = sqlite3.Row, fetchone() returns a Row that supports mapping semantics (column name → value). dict(row) converts that row into an ordinary Python dict. If fetchone() is None, return None instead of calling dict()

* sqlite3.connect(..., check_same_thread=False) => opens (or creates) the DB file; check_same_thread=False allows use from worker threads other than the creating thread (common with ASGI servers). Prefer one connection per request or careful sharing — multiple concurrent writers on one connection still need discipline

* row_factory = sqlite3.Row => rows are addressable by column name (row["email"]) and convert cleanly with dict(row). Default tuple rows would force positional indexing only unless you tracked column order manually


PHASE 20

* bcrypt password hashing => app/auth/password.py wraps the bcrypt library for plaintext never touches the DB
	- hash_password: bcrypt.hashpw(plain.encode("utf-8"), bcrypt.gensalt()).decode("utf-8") — store the returned string in password_hash
	- verify_password: bcrypt.checkpw(plain.encode("utf-8"), hashed.encode("utf-8")) — returns True/False for login

* bcrypt.hashpw => hashes password bytes; the second argument is the salt (here from gensalt, or embedded in an existing hash in other workflows — verification uses checkpw, not re-hashpw)

* bcrypt.gensalt => generates a new random salt per call; bcrypt folds salt + cost into the stored hash string so verification knows how to reproduce the hash

* same plaintext twice => normally different stored strings — each registration runs gensalt(), so two users with the same password get different hashes; reduces rainbow-table value if the DB leaks

* hashpw vs checkpw => hashpw + gensalt when writing password_hash (signup / password change); checkpw when authenticating — extracts salt and params from the stored hash and compares in constant-time style internally — never compare hashes as plain strings yourself

* JWT access tokens => app/auth/tokens.py uses PyJWT (import jwt)
	- jwt.encode: builds a compact signed string from claims (payload) using jwt_secret and jwt_algorithm (settings default HS256)
	- jwt.decode: takes the token string, verifies signature with the same secret and algorithms list; PyJWT validates exp by default so expired tokens fail

* create_access_token(sub, extra=None) => builds payload then encode
	- exp: datetime.now(timezone.utc) + timedelta(minutes=settings.access_token_expire_minutes)
	- base payload: sub, exp, iat — extra dict merged with payload.update(extra) for optional claims without changing the function signature each time

* sub claim => registered JWT claim "subject" — identifies who the token is about (typically users.id from SQLite); stable id preferred over display-only fields

* exp claim => expiry — access tokens should be short-lived so leaked tokens stop working automatically

* iat claim => "issued at" — when the token was minted; optional but useful for auditing, debugging, or rotation policies

* extra claims => examples used in production: email, display_name, roles/scope, jti (JWT id for revocation lists), iss/aud (issuer/audience across services)
	- JWT payload is signed not encrypted — readable by anyone who holds the token — never put secrets there unless using encryption (JWE), which is a different pattern

* decode_access_token(token) => inverse of encode — client sends the token string (not the raw payload); server verifies signature and expiry and returns the claims dict

* why JWTs in production => after login the client presents a bearer proof instead of sending the password on every request
	- typical flow: POST credentials → verify password → create_access_token(sub=user_id, extra=…) → return access_token to client
	- client stores token and sends Authorization: Bearer <token> on API calls
	- dependency/route calls decode_access_token → read sub (and claims) → authorize and serve
	- when exp passes, re-login or use a refresh-token flow (optional second layer not shown in tokens.py alone)

* Bearer and token_type ("bearer") => login responses often include token_type alongside access_token; "Bearer" names the HTTP authorization scheme (RFC 6750), not the JWT format
	- Authorization: Bearer <token> means the credential following the word Bearer is the access token the server issued
	- bearer semantics: whoever possesses the token may use it — possession is treated as proof of identity until exp — no per-request password or extra MAC beyond what is inside the JWT itself
	- treat the token like a password on the client (HTTPS, careful storage); if it leaks, anyone can present it until it expires

* jwt_secret => must stay server-only and long/random in production — whoever knows it can forge tokens; pair with HTTPS because bearer tokens are secrets on the wire

* Auth HTTP schemas => app/auth/schemas.py — Pydantic models for JSON in/out on auth routes (validation, OpenAPI, serialization); separate shapes per operation instead of one mega-model

* RegisterBody => signup body: EmailStr for format-checked email; password with Field(min_length=8, max_length=128); display_name optional with max_length — hashing stays outside the schema

* LoginBody => login body: EmailStr + password — password usually without extra Field rules so failed login responses do not finger-print validation differently from wrong credentials

* TokenResponse => success payload after login/register: access_token (JWT string), token_type defaults to "bearer" for Authorization header usage, user_id so the client knows who logged in without decoding the JWT

* UserPublic => safe user projection for responses (user_id, email, display_name) — intentionally excludes password_hash and password; use instead of returning raw DB rows so secrets never leak by accident

* Auth route handlers => app/api/routes/auth.py — thin HTTP boundary over db.py, password.py, tokens.py, schemas.py
	- APIRouter(prefix="/api/auth", tags=["auth"]) groups endpoints under /api/auth for OpenAPI and imports

* POST register => body validated as RegisterBody; get_user_by_email — if row exists raise HTTPException 409 CONFLICT ("Email already registered") because email is UNIQUE in SQLite
	- otherwise hash_password(body.password), create_user(...), create_access_token(sub=user_id, extra={"email": body.email}), return TokenResponse(access_token=..., user_id=...)
	- success returns JWT immediately so the client can authenticate without a second login call

* POST login => body validated as LoginBody; row = get_user_by_email(body.email)
	- single guard: if no row or not verify_password(plain, row["password_hash"]) raise HTTPException 401 UNAUTHORIZED with generic detail ("Invalid email or password") — avoids revealing whether the email exists
	- otherwise create_access_token(sub=row["id"], extra={"email": row["email"]}), return TokenResponse with user_id row["id"]

* HTTPException in routes => map outcomes to status codes and JSON detail here; keep auth/db and auth/tokens free of FastAPI imports so lower layers stay reusable and testable

* Wiring => app/main.py imports app.api.routes.auth and app.include_router(auth.router) alongside chat, documents, health so auth endpoints are live on the same FastAPI app


PHASE 21

* FastAPI dependencies sync vs async => Depends(...) accepts either def or async def; choose based on what the dependency does, not “everything async must be async def”

* Phase plan pattern => app/auth/deps.py helpers such as get_current_user are written as plain def in the roadmap — decode JWT + stdlib sqlite3 lookup — that matches intentional sync deps

* sync def dependency + async def route => FastAPI runs blocking sync dependencies in a thread pool when the endpoint is async, so sqlite3.get_user_by_id inside get_current_user does not occupy the asyncio event loop

* async def dependency => right choice when the dependency itself awaits async I/O — async database drivers (asyncpg, SQLAlchemy async session), httpx.AsyncClient, async Redis clients — keeps one consistent async stack

* async def dependency calling blocking sqlite3 directly => still blocks the event loop inside the coroutine — worse than sync def + pool unless work is offloaded (executor / sync helper called correctly)

* rule of thumb => blocking sqlite + PyJWT decode → sync dependency callable is appropriate until user lookup becomes truly async

* Phase 21 wiring => get_current_user in app/auth/deps.py reads Authorization: Bearer <token>, verifies JWT via decode_access_token, resolves sub to SQLite row via get_user_by_id, raises 401 on missing header/wrong scheme/invalid exp/sub/missing user — this is the enforcement choke point for “who is calling”

* GET /api/auth/me => depends on get_current_user and returns UserPublic (user_id, email, display_name) — lets a client confirm identity from the token alone without trusting any user id in the request body

* Route order and /me => /me does not need to be declared above /login or /register just because it is a literal path — FastAPI matches distinct paths independently. Order only matters when a parameterized route could steal a literal (e.g. define /users/me before /users/{user_id}); no such conflict on the current auth router layout

* Chat security flow => POST /api/chat uses current_user_optional: same Bearer extraction pattern but returns None if header absent or token/user invalid — pipeline still runs when anonymous mode allows it

* settings.allow_annonymous_chat => when false and creds_user is None → HTTP 401 (“login required”) — anonymous callers cannot hit chat at all. When true → trusted_user_id may be None — demo / legacy behavior without a JWT

* trusted_user_id => taken only from creds_user["id"] after successful decode + DB lookup — intentionally ignores ChatRequest.user_id for authorization and personalization so clients cannot impersonate another user by forging JSON

* End-to-end JWT flow for chat => client obtains access_token from register/login → sends Authorization: Bearer <access_token> on /api/chat → optional dependency validates and loads user → run_rag_pipeline receives server-derived user_id only → profiles and telemetry tied to real identity when anonymous mode is off


PHASE 22 (NiceGUI auth + asyncio lesson)

* NiceGUI auth UX => login form on `/` when auth is required and no valid session; loopback HTTP to `/api/auth/login` and `/api/auth/register`; `verify_stored_bearer()` calls `/api/auth/me` with stored Bearer token so a valid JWT skips the login UI; `/register` shows a post-success “You're signed in” + “Go to chat”; `/login` remains for anonymous-demo mode (`allow_annonymous_chat`) only — stored session uses `app.storage.user` (access_token, user_id, email)

* Why the server “froze” on login/register => Uvicorn drives an asyncio event loop; NiceGUI `on_click` and synchronous page setup run work on that loop. Using **`httpx.Client`** (`with Client(...)`) performs **blocking** socket I/O — the loop cannot process other requests or UI updates until the HTTP call returns — looks like a global freeze with symptoms worst on same-process loopback to the same app

* Fix => **`httpx.AsyncClient`** created once in **FastAPI lifespan** (`app.state.internal_http_client`), **`base_url=http://127.0.0.1:{app_port}`**, **`timeout` configured**, **`await client.aclose()`** in **`finally`** after `yield` — NiceGUI paths use **`async def`** pages/handlers and **`await http().get/post(...)`** so I/O yields and the server stays responsive

* Relative paths => internal calls use paths like `/api/auth/me` against the client `base_url` — no duplicated URL building

* Related caveat => **`run_rag_pipeline`** inside **`async def send()`** is still **synchronous** heavy work — long RAG calls can block the loop until offloaded (e.g. `asyncio.to_thread`) — separate from the auth HTTP freeze but worth knowing for chat latency under load


Architecture decisions — auth, UI, ops

* Two ways chat sees `user_id` => **`POST /api/chat`** derives **`trusted_user_id`** only from a verified Bearer JWT (`current_user_optional` → SQLite row). **NiceGUI** skips that route and calls **`run_rag_pipeline(..., user_id=app.storage.user["user_id"])`** in-process after the UI auth flow — identity on the UI path is **whatever the server stored** after successful login/register, not a second JWT verification per message — API clients stay JWT-bound; UI trusts server-side session storage once populated

* NiceGUI uses loopback HTTP for auth => register/login/me hit the same FastAPI routes via **`AsyncClient`** rather than importing **`hash_password` / `create_access_token`** inside **`ui.py`** — keeps **one behavioral contract** for credentials and tokens at the cost of an extra HTTP hop and dependence on **`127.0.0.1:{app_port}`** being reachable from inside the process/container

* JWT in NiceGUI **`app.storage.user`** => convenient for a demo SPA-style UI; storage is **not HttpOnly-cookie isolated** — mitigate XSS and treat token leakage like password leakage; production hardening often moves toward http-only cookies + CSRF strategy or very short-lived access tokens (see roadmap Phase 24)

* **`allow_annonymous_chat`** => explicit **product / ops flag**: disables anonymous **`/api/chat`** when false; NiceGUI uses it to decide whether **`/`** requires login — document env alignment (**`ALLOW_ANNONYMOUS_CHAT`** matches **`allow_annonymous_chat`** field spelling in Settings) so prod never accidentally runs demo-open chat

* Docker dependency reproducibility => image builds should **`COPY pyproject.toml` + `uv.lock`** before **`uv sync`** so the container matches the committed lockfile; after adding deps (**PyJWT**, **bcrypt**, etc.) **`docker compose build --no-cache app`** (or service name) avoids stale layers that omit packages and surface **`ModuleNotFoundError`** at runtime
