# סיכום הפרויקט — RAG From Scratch

## מהו הפרויקט?

**RAG From Scratch** הוא פרויקט פורטפוליו המדגים בנייה של מערכת RAG
ברמת Production, לצד פלטפורמת למידה אדפטיבית המסייעת למשתמשים ללמוד את
עקרונות ה־RAG באופן מעשי והדרגתי.

המערכת אינה רק צ'אט שמחזיר תשובות מתוך מסמכים. היא משלבת:

- קליטה, חלוקה ואינדוקס של מסמכים.
- חיפוש סמנטי במאגר וקטורי, עם מנגנון גיבוי מבוסס BM25.
- יצירת תשובות בעזרת מודל שפה והקשר שנשלף ממאגר הידע.
- הערכה אוטומטית של הידע שמפגין המשתמש במהלך השיחה.
- פרופיל למידה אישי הכולל רמת שליטה, חוזקות ופערי ידע.
- התאמת התשובות, השאלות ורמת ההסבר לרמת המשתמש.
- מסלול לימוד מובנה הכולל שאלות פתוחות, שאלות רב־ברירה ושערי התקדמות.
- הזרמת תשובות בזמן אמת לממשק המשתמש.

זרימת העבודה המרכזית מנוהלת כגרף:

```text
אחזור מידע → יצירת תשובה → הערכת המשתמש → עדכון הפרופיל → סיום
```

## היכולות המרכזיות

- צ'אט RAG אדפטיבי עם זיכרון ברמת session.
- הרשמה, התחברות ואימות באמצעות JWT.
- העלאת מסמכים ואינדוקס שלהם למאגר הידע.
- Onboarding ואבחון ראשוני של רמת המשתמש.
- מעקב אחר התקדמות לפי נושאים ושלבי לימוד.
- תמיכה בשאלות פתוחות ובשאלות אמריקאיות.
- התאמת prompt לחמש רמות ידע, ממתחיל ועד מומחה.
- תמיכה ב־OpenAI כספק ראשי וב־Ollama כחלופה מקומית.
- מנגנוני graceful degradation, cache ו־circuit breaker.
- ניטור, מדדים ולוגים מובנים לסביבת Production.

## הסטאק הטכנולוגי

### שפה ותשתית Backend

- **Python 3.11+**
- **FastAPI** לבניית ה־API.
- **Uvicorn** כשרת ASGI.
- **Pydantic** להגדרת schemas, ולידציה והגדרות סביבה.
- **SQLite** לשמירת משתמשים ופרופילי למידה, עם נתיב מעבר עתידי ל־PostgreSQL.
- **Redis** לשכבת cache.

### ממשק משתמש

- **NiceGUI** לבניית ממשק הצ'אט, מסכי האימות, תהליך ה־onboarding ולוח ההתקדמות.
- **Server-Sent Events (SSE)** להזרמת tokens ותוצאות סופיות בזמן אמת.

### AI ו־RAG

- **LangChain** לבניית רכיבי שרשרת ה־RAG.
- **LangGraph** לניהול תהליך העבודה כ־state machine.
- **OpenAI** כספק מודלי השפה הראשי.
- **Ollama** להפעלה מקומית ולגיבוי.
- **Sentence Transformers** ליצירת embeddings מקומיים.
- **ChromaDB** כמאגר וקטורי ראשי.
- **BM25** כמנגנון אחזור חלופי כאשר החיפוש הווקטורי אינו זמין.
- **Tiktoken** למדידת tokens.

### אבטחה

- **JWT** לניהול authentication.
- **bcrypt** לגיבוב סיסמאות.
- הגבלת נתיבי קבצים בתהליך קליטת מסמכים.
- שמירת secrets במשתני סביבה בלבד.
- allowlist לפעולות SQL דינמיות.

### בדיקות, תפעול ו־Production

- **Pytest** ו־**pytest-asyncio** לבדיקות.
- **Docker** ו־**Docker Compose** להרצת המערכת והשירותים התומכים.
- **Nginx** כ־reverse proxy בסביבת Production.
- **Prometheus** ו־**Grafana** למדדים ולוחות ניטור.
- **ELK** לריכוז וניתוח לוגים.
- יעד הפריסה המתועד הוא **AWS EC2**.

## מבנה כללי של הקוד

```text
src/app/                 אפליקציית FastAPI, ה־UI, אימות ופרופילי משתמש
src/app/api/routes/      נתיבי API לצ'אט, מסמכים, משתמשים ו־onboarding
src/agents/              גרף הסוכנים, state, nodes ומנגנוני הערכה
src/rag/                 אחזור, embeddings, indexing, ספקי LLM ועמידות
knowledge-base/          תוכנית הלימוד, שאלות, rubrics ושערי התקדמות
tests/                   בדיקות יחידה ואינטגרציה
monitoring/              הגדרות ולוחות ניטור
commit-specs/            מפרטים ממוקדים לכל שלב פיתוח
```

## תהליך העבודה עם הסוכנים

1. העבודה מחולקת ל־commits קטנים, כאשר כל commit מטפל בנושא אחד בלבד.
2. Claude קורא את מפרט ה־commit ואת מצב הפרויקט ובוחר את הסוכן המתאים.
3. הסוכן מקבל רק את ההקשר הנדרש למשימה: זהות, מצב נוכחי, מפרט ו־handoffs רלוונטיים.
4. הסוכן עובד רק בתחום ובקבצים שבבעלותו.
5. בעיה בתחום אחר נרשמת כ־cross-domain finding ומוחזרת ל־Claude לניתוב.
6. בסיום הפיתוח מופעלות בקרות האיכות הנדרשות לפי סוג השינוי.
7. ממצאים מוחזרים לסוכן המטפל עד שכל החסימות נפתרות.
8. ראש הצוות, Eran Mani, נותן אישור סופי לפני כל commit.
9. לאחר האישור מתעדכנים מצב הפרויקט, יומן ההחלטות והמסמכים הרלוונטיים.

כל התקשורת בין הסוכנים עוברת דרך Claude. סוכנים אינם פונים ישירות זה לזה
ואינם משנים קבצים מחוץ לתחום שלהם. כך נשמרים עקביות, אחריות ברורה, יכולת
ביקורת והיסטוריית שינויים שקל להבין ולהחזיר לאחור.

## עקרונות הפיתוח

- בעלות ברורה על כל תחום וקובץ.
- commit אחד לכל נושא.
- אישור אנושי לפני כל commit.
- בקרות איכות בהתאם לסיכון ולסוג השינוי.
- החלטות ארכיטקטוניות מתועדות ב־`DECISIONS.md`.
- מצב הפרויקט נשמר ב־`project-state.json`.
- העבודה המתוכננת מוגדרת ב־`commit-protocol.md` וב־`commit-specs/`.
- חבילות ההקשר מוגבלות כדי לחסוך tokens ולמנוע עומס מידע.
- ניתן להריץ משימות במקביל רק כאשר אין ביניהן תלות או חפיפה בקבצים.
